[h1] Chapter 1. Introduction
[h2] 1.1. Motivation
[h4] The motorcycle industry faces significant digital fragmentation that creates inefficiencies for both consumers and service providers. Research indicates that 81% of motorcycle parts consumers favor specialty websites such as RevZilla and Cycle Gear for their shopping needs, while 75% also choose traditional local dealerships as their go-to for parts purchases [22]. This demonstrates that consumers currently must navigate multiple disconnected platforms to complete basic ownership activities: separate websites for purchasing motorcycles, booking maintenance services, and engaging with community forums.
[h4] Content analysis of motorcycle forums and social media platforms reveals that word-of-mouth and peer recommendations play a crucial role in shaping brand perceptions and influencing purchasing decisions [22]. However, this fragmentation results in poor user experience and missed opportunities for integrated commerce and service delivery.
[h4] The U.S. motorcycle market size was valued at USD 10.48 billion in 2024 and is expected to expand at a CAGR of 4.1% from 2025 to 2030 [23]. This growth trajectory indicates significant market opportunity, yet industry revenue has declined at a CAGR of 2.2% over the past five years, reaching an estimated $6.7 billion in 2025 [24], suggesting that traditional business models face challenges that digital integration could address.
[h4] From a technical perspective, this project addresses the software engineering challenge of creating a scalable, multi-tenant platform that can integrate diverse stakeholders (manufacturers, dealers, service providers, and consumers) while maintaining performance and security standards. The solution requires sophisticated system architecture design, database optimization for complex multi-entity relationships, and user interface design that serves multiple user personas with varying technical expertise.
[h4] Recent consumer sentiment surveys indicate that 67% of potential powersports buyers believe the economy will improve or remain stable, and 53% feel the economic climate doesn't influence their purchasing decisions [25]. Additionally, 65% anticipate buying a new ride by summer, with a majority confident in a swift acquisition process [25]. This data suggests strong market demand that an integrated platform could more effectively serve.
[h4] The business motivation centers on addressing documented market inefficiencies. Consumer research identifies inventory issues as the main deterrent to purchases (25%), followed by pricing concerns (29%) and long wait times (12%) [26]. An integrated platform could potentially address these controllable factors through better inventory visibility and streamlined communication processes.
[h4] This capstone project provides an opportunity to apply software engineering principles to documented market problems while demonstrating competency in full-stack development, system architecture design, and user experience optimization. The project's scope allows for exploration of contemporary technologies including cloud-based microservices architecture, responsive web design, and database optimization techniques.
[h4] The research contributes to understanding how digital platforms can transform traditional industries by reducing transaction costs, improving information flow, and creating new value propositions for multiple stakeholder groups simultaneously, addressing the documented fragmentation in motorcycle commerce and services.
[h2] 1.2. Project Overview
[h4] This phase focuses on the implementation of MotoMarket, an integrated web platform that unifies motorcycle discovery and commerce, service booking, and community interaction within a coherent technical architecture. Building upon the conceptual models, requirements, and interface designs produced in Phase 1, Phase 2 delivers a functional prototype that operationalizes the chosen stack—Java Spring Boot for backend services, SQL Server for persistence, Cloudinary for media handling, and React/Vite for a responsive single-page frontend—while adhering to principles of security, maintainability, and scalability.
[n] Project Scope and Deliverables:
[h4] Implement a working end-to-end system encompassing authentication, catalog management, orders, service bookings, forum interactions, media workflows, and administrative oversight.
[h4] Provide documented REST endpoints, role-based access controls, and a seeded dataset for demonstration and evaluation.
[h4] Deliver a responsive frontend with guarded routes, consistent interaction patterns, and basic error handling aligned to development needs.
[n] Core Implementation Activities:
[h4] Backend Engineering: Develop Spring Boot services with stateless JSON Web Token (JWT) authentication and role-based authorization; model domain entities (User, Motorcycle, Order, Review, TestRide, MaintenanceService, Store, Staff, ServiceOffering); implement repositories and services using JPA/Hibernate; configure Cloudinary for image storage and retrieval; provision initial data via a seeding mechanism.
[h4] Frontend Engineering: Implement SPA routing with React Router; enforce protected navigation for ADMIN , STAFF , and CUSTOMER roles; integrate Axios with token interception and authentication fallback; manage lightweight state with Zustand; apply TailwindCSS for consistent, responsive UI.
[h4] Integration and Media Handling: Support multi-image upload, replacement, and deletion flows; ensure contract alignment between client and server modules; validate basic cross-module workflows (e.g., discovery → order intent → service scheduling).
[n] System Components Implemented:
[h4] Authentication and Authorization: Registration and login with JWT issuance; role-based guards across server endpoints and client routes.
[h4] Motorcycle Catalog: Listing, detail views, search/filtering, comparison; administrative create/update/delete and image management.
[h4] Orders: Order creation, status tracking, user-specific retrieval; administrative listing and status adjustments; placeholder payment recording.
[h4] Service Bookings: Test ride and maintenance service scheduling; user histories; store discovery and nearest-store computation endpoints; recent services and statistics.
[h4] Community Features: Forum posts and comments; like and report interactions; review modeling with moderation flags.
[h4] Administrative Dashboard: Summary statistics, recent orders, low-stock detection; user and order management; staff assignment for test rides.
[n] Validation and Evaluation:
[h4] Functional validation conducted in a development environment, confirming end-to-end operability across primary user journeys (authentication, catalog interactions, order intent, service scheduling, and forum engagement).
[h4] Verification emphasized correctness of role enforcement, data persistence, and media handling; UI responsiveness and basic error states were inspected for coherence.
[n] Limitations and Deferred Features:
[h4] Email Verification: Signup currently proceeds without email confirmation; identity assurance to be introduced via verification tokens and activation workflows.
[h4] Notifications: No real-time or email notifications for state changes (orders, bookings, forum interactions); a notification service with user preferences will be added.
[h4] Direct Payments: Payment processing is limited to recording transaction identifiers; gateway integration (e.g., Stripe/PayPal/VNPay) and webhook reconciliation will be implemented.
[h4] Map Functionality: The interactive map-based store locator is under implementation and not yet operational; completion is planned in Phase 3.
[h4] Operational Readiness: Deployment automation, observability (structured logs, metrics, tracing), security hardening (stricter CORS, input validation, rate limiting), and performance tuning (caching, indexing, image optimization) are scheduled for Phase 3.
[n] Project Boundaries and Future Phases:
[h4] Phase 2 is bounded to core functionality and development validation; it does not include production deployment or comprehensive observability.
[h4] Phase 3 will complete deferred capabilities (email verification, notifications, payment gateway, map UI), implement deployment pipelines and environment-based secret management, strengthen security posture, and refine user experience through systematic testing and measurement.
[n] Academic Learning Objectives:
[h4] Demonstrate application of software engineering principles in secure API design, role-based authorization, domain modeling, and media integration.
[h4] Apply SPA architecture patterns and client-side state management; evaluate maintainability and scalability under realistic development constraints.
[h4] Translate Phase 1 specifications and prototypes into an operational system while preserving traceability between requirements, models, and implementation.
[n] Research Contribution:
[h4] Provide a practical instantiation of an integrated multi-stakeholder platform for a digitally fragmented industry, showing how unified workflows reduce transaction costs and improve information flow.
[h4] Offer an implementation blueprint and design patterns that are adaptable to other specialized markets requiring coordinated commerce, service delivery, and community features.

[h1] Chapter 3. Proposed Approach, Analysis, Design, Implementation
[h2] 3.1. Analysis Phase
[h4] Phase 2 focused on validating the conceptual artefacts developed in Phase 1 through operational implementation, empirical testing, and iterative requirements refinement. The analysis targeted the persistent fragmentation of the motorcycle ecosystem by examining how integrated workflows—discovery, transaction intent (orders), service coordination (test rides and maintenance), and community engagement—behave under a unified architecture. Stakeholder needs were revisited for three principal groups: motorcycle enthusiasts seeking trustworthy information exchange, service providers requiring predictable scheduling and communication channels, and buyers/sellers needing reliable catalogues and transaction support. This validation converted Phase 1’s high-level objectives into implementable epics and measurable non-functional targets (security, performance, scalability, maintainability).
[h4] Requirements were refined into a prioritized backlog comprising authentication and role-based access control, motorcycle catalogue functions (search, comparison, media management), orders with status transitions, service bookings, forum interactions, and administrative oversight. Non-functional specifications were elaborated to include stateless JWT-based authentication, explicit authorization rules, CORS configuration for client–server interaction, and media offloading to Cloudinary to reduce storage and bandwidth constraints. Data modelling consolidated multi-entity relationships (e.g., User–Order–Motorcycle , User–MaintenanceService , Motorcycle–Review ) to support transactional integrity while accommodating extensibility for moderation and analytics. Empirical evaluation in the development environment verified end-to-end operability and surfaced controlled gaps: email verification during signup, user notifications for state changes (orders, bookings, forum interactions), direct online payment processing via a gateway, and an interactive map-based store locator. These gaps were documented for Phase 3, together with deployment, observability, and UX enhancements.
[h2] 3.2. Design Phase
[h4] The design work in Phase 2 translated refined requirements into a concrete, maintainable implementation aligned with the three-tier architecture envisioned in Phase 1. The presentation layer is realized as a React/Vite single-page application employing TailwindCSS, Axios, React Router, and a lightweight state store; the business logic layer is implemented in Spring Boot with stateless JWT authentication and role-based authorization; and the data access layer uses SQL Server via JPA/Hibernate. This separation of concerns supports scalability, fault isolation, and disciplined evolution of features.
[h4] Design artefacts were updated to reflect implementation decisions and ensure traceability from Phase 1 models to deployed components. API contracts and controller boundaries were specified to expose catalogue, orders, bookings, forum, image upload, and administrative endpoints; security design codified public routes and role-restricted actions; error handling and token interception patterns were defined for robust client–server communication. Domain services encapsulate core logic for entity lifecycles and state transitions, while repositories provide transactional persistence and query capabilities. Media workflows—multi-image upload, replacement, deletion—were designed to operate through Cloudinary-backed endpoints, reducing infrastructure overhead and standardizing asset management.
[h4] On the frontend, routing and guarded navigation enforce role-based access to administrative and staff areas, with consistent patterns for authentication fallback and basic error states. Component and style guidelines were derived from Phase 1’s Figma prototypes and implemented using TailwindCSS to achieve responsive layouts and visual coherence. Sequence and use case diagrams were extended to cover implemented flows (registration/login, catalogue browsing and comparison, order creation and status updates, booking schedules and histories, forum posting and moderation signals), while class diagrams captured object relationships and constraints as realized in code.
[h4] The design explicitly accommodates deferred capabilities required for production readiness in Phase 3: email verification (verification tokens and activation flows), notification services (real-time push via WebSocket/SSE and out-of-band email/SMS), payment gateway integration with secure webhook reconciliation, and an interactive map-based store locator. Operational design considerations—deployment automation, environment-based secret management, observability (structured logs, metrics, tracing), security hardening (stricter CORS, input validation, rate limiting), and performance tuning (caching, indexing, image optimization)—were planned to support reliable and scalable operation beyond the development environment.

[h4] An addition to the Spring framework, Spring Boot streamlines the process of creating enterprise Java apps. [15] It makes it simpler to develop standalone, production-grade Spring-based apps by offering auto-configuration, embedded servers, and production-ready features right out of the box.

[n] 2. Why Spring Boot for MotoMarket
[h4] Spring Boot was chosen as the backend framework for the following reasons:

[h4] Database Integration: Spring Data JPA integration simplifies database operations and provides object-relational mapping capabilities, making it easier to manage complex relationships between users, motorcycles, service providers, and forum data.
