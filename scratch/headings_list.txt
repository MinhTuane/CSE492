2. [h1] ABSTRACT 
3. [h4] The motorcycle retail and after‑sales services market remains fragmented across marketplaces, dealer networks, independent repair shops, and community forums, producing disjointed user journeys and elevated transaction costs. This Phase 2 report presents the implementation of MotoMarket, an integrated, web‑based platform designed to unify discovery, commerce, service booking, and community interaction within a cohesive technical architecture. Phase 1 established the project’s conceptual and methodological foundation through requirements analysis and the development of system artifacts (use case, sequence, and class diagrams), alongside high‑fidelity user interface prototypes. It also selected a technology stack comprising React for frontend development, Java Spring Boot for backend services, SQL Server for relational persistence, and Cloudinary for media handling.
4. [h4] Building upon these artifacts, Phase 2 delivers a functional prototype that operationalizes the chosen architecture. The backend employs stateless JSON Web Token (JWT) authentication with role‑based access control and JPA/Hibernate for data management, while the frontend implements a single‑page application using React/Vite, TailwindCSS, Axios, React Router, and a lightweight state store. Implemented modules include user authentication; a motorcycle catalog with search, comparison, and image workflows; order management with status tracking; service booking for test rides and maintenance (including store discovery, nearest‑store computation, recent services, and statistics); community features for posts and comments with like/report interactions; and an administrative dashboard for operational oversight. The system functions end‑to‑end in a development environment, validating the feasibility and maintainability of the integrated approach.
5. [h4] Several capabilities are deliberately deferred to subsequent work. Email verification during signup has not yet been implemented; user notifications for state changes (orders, bookings, and community interactions) are currently absent; and payment processing is limited to recording transaction identifiers without direct gateway integration. In particular, the map‑based functionality (interactive store locator and nearest‑store experience) is under implementation and not yet operational. These features, together with full integration, deployment automation, observability, security hardening, and user experience refinement, are planned for Phase 3, which will transition MotoMarket from a validated prototype to a production‑ready platform.

156. [h1] Chapter 3. Proposed Approach, Analysis, Design, Implementation
157. [h2] 3.1. Analysis Phase
158. [h4] Phase 2 focused on validating the conceptual artefacts developed in Phase 1 through operational implementation, empirical testing, and iterative requirements refinement. The analysis targeted the persistent fragmentation of the motorcycle ecosystem by examining how integrated workflows—discovery, transaction intent (orders), service coordination (test rides and maintenance), and community engagement—behave under a unified architecture. Stakeholder needs were revisited for three principal groups: motorcycle enthusiasts seeking trustworthy information exchange, service providers requiring predictable scheduling and communication channels, and buyers/sellers needing reliable catalogues and transaction support. This validation converted Phase 1’s high-level objectives into implementable epics and measurable non-functional targets (security, performance, scalability, maintainability).
159. [h4] Requirements were refined into a prioritized backlog comprising authentication and role-based access control, motorcycle catalogue functions (search, comparison, media management), orders with status transitions, service bookings, forum interactions, and administrative oversight. Non-functional specifications were elaborated to include stateless JWT-based authentication, explicit authorization rules, CORS configuration for client–server interaction, and media offloading to Cloudinary to reduce storage and bandwidth constraints. Data modelling consolidated multi-entity relationships (e.g., User–Order–Motorcycle , User–MaintenanceService , Motorcycle–Review ) to support transactional integrity while accommodating extensibility for moderation and analytics. Empirical evaluation in the development environment verified end-to-end operability and surfaced controlled gaps: email verification during signup, user notifications for state changes (orders, bookings, forum interactions), direct online payment processing via a gateway, and an interactive map-based store locator. These gaps were documented for Phase 3, together with deployment, observability, and UX enhancements.
160. [h2] 3.2. Design Phase
161. [h4] The design work in Phase 2 translated refined requirements into a concrete, maintainable implementation aligned with the three-tier architecture envisioned in Phase 1. The presentation layer is realized as a React/Vite single-page application employing TailwindCSS, Axios, React Router, and a lightweight state store; the business logic layer is implemented in Spring Boot with stateless JWT authentication and role-based authorization; and the data access layer uses SQL Server via JPA/Hibernate. This separation of concerns supports scalability, fault isolation, and disciplined evolution of features.
162. [h4] Design artefacts were updated to reflect implementation decisions and ensure traceability from Phase 1 models to deployed components. API contracts and controller boundaries were specified to expose catalogue, orders, bookings, forum, image upload, and administrative endpoints; security design codified public routes and role-restricted actions; error handling and token interception patterns were defined for robust client–server communication. Domain services encapsulate core logic for entity lifecycles and state transitions, while repositories provide transactional persistence and query capabilities. Media workflows—multi-image upload, replacement, deletion—were designed to operate through Cloudinary-backed endpoints, reducing infrastructure overhead and standardizing asset management.
163. [h4] On the frontend, routing and guarded navigation enforce role-based access to administrative and staff areas, with consistent patterns for authentication fallback and basic error states. Component and style guidelines were derived from Phase 1’s Figma prototypes and implemented using TailwindCSS to achieve responsive layouts and visual coherence. Sequence and use case diagrams were extended to cover implemented flows (registration/login, catalogue browsing and comparison, order creation and status updates, booking schedules and histories, forum posting and moderation signals), while class diagrams captured object relationships and constraints as realized in code.
164. [h4] The design explicitly accommodates deferred capabilities required for production readiness in Phase 3: email verification (verification tokens and activation flows), notification services (real-time push via WebSocket/SSE and out-of-band email/SMS), payment gateway integration with secure webhook reconciliation, and an interactive map-based store locator. Operational design considerations—deployment automation, environment-based secret management, observability (structured logs, metrics, tracing), security hardening (stricter CORS, input validation, rate limiting), and performance tuning (caching, indexing, image optimization)—were planned to support reliable and scalable operation beyond the development environment.

221. [h2] 4.1. Architectural Overview
222. [h4] The MotoMarket platform in Phase 2 is implemented using a layered, three‑tier architecture consisting of a presentation layer, a business logic layer, and a data access layer. This separation of concerns is intended to support scalability, maintainability, and clear allocation of responsibilities across the system.
223. [h4] The presentation layer is realized as a single‑page web application. It is responsible for rendering user interfaces, handling client‑side navigation, and orchestrating API requests to the backend. The business logic layer is implemented as a set of RESTful services that encapsulate domain rules for authentication, motorcycle catalog management, orders, bookings, community features, and administrative operations. The data access layer manages persistence and retrieval of domain entities in a relational database.
224. [h4] The system follows a stateless communication model between client and server, using JSON Web Tokens (JWT) for authentication and role‑based authorization for access control. Media assets, particularly motorcycle images and user‑generated content, are offloaded to a cloud‑based media service, which reduces load on the core application and simplifies asset management. This overall architecture reflects the conceptual design developed in Phase 1 while adapting it to practical implementation constraints.
225. [h2] 4.2. Backend Design
226. [h4] The backend is designed as a modular REST API that exposes clear boundaries between functional areas. Each module is implemented as a set of controllers, services, and repositories, following a standard pattern:
227. [h4] The authentication module manages user registration and login, issues JWTs, and enforces role‑based access.
228. [h4] The catalog module provides endpoints for browsing, searching, and comparing motorcycles, as well as administrative endpoints for creating, updating, and deleting motorcycle records and associated images.
229. [h4] The orders module supports order creation, retrieval, and status updates, along with a placeholder endpoint for recording payment transactions.
230. [h4] The bookings module manages test ride and maintenance service scheduling, retrieval of user booking histories, and access to store information, recent services, and service statistics.
231. [h4] The forum module enables listing, viewing, creating, and interacting with posts and comments, including like and report actions.
232. [h4] The administration module aggregates management capabilities for motorcycles, orders, users, and test rides, as well as dashboard statistics and staff assignment.
233. [h4] Each controller delegates core logic to dedicated service classes, which encapsulate validation and domain rules, and to repository classes, which perform database operations. This design reduces coupling between the API surface and persistence, and simplifies testing and future evolution. Cross‑cutting concerns such as error handling and logging are handled centrally where appropriate.
234. [h4] Authentication and authorization are enforced through a JWT validation filter and security configuration that define which endpoints are public and which require authenticated access with specific roles (e.g., customer, staff, administrator). The backend remains stateless with respect to user sessions, relying exclusively on tokens presented by the client.
235. [h2] 4.3. Data Model Design
236. [h4] The data model builds on the conceptual class diagrams produced in Phase 1 and is refined to support the implemented features in Phase 2. Core entities include:
237. [h4] A user entity that represents individual accounts, with attributes for identity, contact information, role, and activity status, and relationships to orders, bookings, and reviews.
238. [h4] A motorcycle entity that captures brand, model, technical specifications, pricing, stock levels, and associated images and reviews.
239. [h4] An order entity that groups one or more motorcycles into a transaction, with fields for amounts, payment method, status, and relevant timestamps in the order lifecycle.
240. [h4] Review, test ride, maintenance service, staff, store, and service offering entities that support community feedback, booking flows, service management, and staff assignment.
241. [h4] Relationships between entities (for example, between users and their orders, between motorcycles and their reviews, or between services and stores) are explicitly modeled to support integrated operations across modules. The design aims to balance normalization with query performance for the expected access patterns, such as listing a user’s orders, retrieving all services for a given motorcycle, or aggregating statistics for administrative dashboards.

423. [h2] 4.7. Discussion 
424. [h3] 4.7.1. Implementation Success and Key Deliverables
425. [h4] Phase 2 successfully delivered a working integrated platform with 32 motorcycles spanning 10 major brands, demonstrating that the chosen technology stack can support the complete user journey from discovery through transaction and service booking. The most significant achievement is proving that fragmented motorcycle ecosystem functions—browsing catalogs, placing orders, scheduling services, and community discussion—can operate cohesively within a single platform architecture.
426. [h4] The Cloudinary integration solved what could have been a major development bottleneck. Instead of building custom image storage and delivery infrastructure, the platform leverages professional-grade media management with automatic optimization and CDN delivery. This freed development time to focus on core business logic while delivering better image performance than a custom solution would achieve.
427. [h3] 4.7.2. Technical Decisions That Shaped Implementation
428. [h4] Stateless Authentication Approach: JWT-based authentication eliminates server-side session storage, simplifying horizontal scaling for future growth. However, this creates token management challenges—once issued, tokens remain valid until expiration regardless of account changes. During development, testing administrative actions required waiting for token expiration or manually clearing browser storage rather than simply invalidating sessions. For production, implementing token refresh mechanisms and shorter token lifetimes will balance security against user experience.
429. [h4] Order Status Workflow Design: Orders progress through defined status transitions (PENDING → PAID → PROCESSING → SHIPPED → DELIVERED) with validation preventing invalid jumps between states. This ensures inventory doesn't decrement for unpaid orders, but requires careful coordination between payment confirmation and fulfillment. The decision to update stock only when orders reach DELIVERED status protects against phantom inventory depletion from abandoned orders, though it creates brief windows where popular motorcycles could be oversold if multiple customers order simultaneously.
430. [h4] Dual Image Upload Strategy: Supporting both pre-uploaded URLs and direct file uploads during motorcycle creation accommodates different administrative workflows. Quick catalog updates can use pre-uploaded images, while detailed curation can handle uploads during creation. This flexibility proved essential during testing when bulk catalog updates required different approaches than individual motorcycle creation. However, managing image cleanup when database operations fail required explicit rollback logic to prevent orphaned Cloudinary resources.
431. [h4] Client-Side Catalog Filtering: The motorcycle catalog implements filtering and sorting entirely on the frontend after loading all motorcycles. This enables instant filter changes without API calls and simplified initial development, but doesn't scale beyond several hundred motorcycles. With 32 seeded motorcycles, the approach performs well, but migrating to server-side filtering and pagination will be necessary before production launch with larger inventories.
432. [h3] 4.7.3 Real Challenges Encountered
433. [h4] Status Transition Validation Complexity: Initial implementation allowed arbitrary order status changes, which during testing led to scenarios where orders marked as delivered without payment triggered inventory depletion. Implementing explicit transition rules required mapping valid next states for each current status—PENDING orders can become PAID or CANCELLED, but not directly DELIVERED. This state machine logic prevents business rule violations while still allowing administrative corrections for edge cases.
434. [h4] Multi-Image Management Workflow: Managing multiple images per motorcycle—adding new images, removing specific images, replacing all images—required coordinating Cloudinary operations with database updates. Early attempts treated these as separate operations, leading to inconsistencies when one operation succeeded but the other failed. The solution involved transaction-like behavior where Cloudinary uploads complete before database persistence, with cleanup logic when database operations fail.
435. [h4] Service Booking Location Data: Managing store information and making it accessible for test ride and maintenance service bookings required careful consideration of data structure. The current implementation stores basic store information that users can select during booking, with plans for enhanced location-based features in Phase 3.
436. [h4] Filter Reset Expectations: Users applying multiple filters expected a single "clear all" action to reset everything simultaneously. Initial implementation required clearing each filter individually, creating friction in the user experience. Adding comprehensive reset functionality required coordinating state updates across multiple filter inputs while ensuring the catalog re-filtered correctly after clearing.
437. [h3] 4.7.4. What Was Strategically Deferred
438. [h4] Email Verification: User registration proceeds without email confirmation, creating security vulnerabilities and account recovery challenges. Implementing email verification requires SMTP server configuration, token generation, verification email templates, and activation workflow—representing 1-2 weeks of work. Since Phase 2 prioritized validating integrated platform mechanics, accepting unverified accounts for prototype testing was reasonable. Production deployment absolutely requires email verification before launch.
439. [h4] Payment Gateway Integration: Orders record payment methods and transaction identifiers but don't process actual payments through providers like Stripe or VNPay. Real payment integration introduces PCI compliance requirements, webhook handling for asynchronous confirmations, refund processing, and financial reconciliation—complexity beyond Phase 2's validation scope. The placeholder approach proves order workflow correctness while clearly marking the integration boundary for Phase 3.
440. [h4] Real-Time Notifications: The system lacks notifications for order status changes, booking confirmations, or forum replies. Users must manually check for updates rather than receiving proactive alerts. Building notification infrastructure requires message templating, delivery mechanisms (email/SMS/push), user preference management, and tracking sent notifications. While notifications significantly improve engagement, their absence doesn't prevent validating core functionality.
441. [h4] Interactive Map and Location Features: Store location functionality with geospatial calculations and interactive map interfaces for visualizing store locations, route planning, and nearest-store identification remains to be implemented in Phase 3. The current system includes basic store data management that will serve as the foundation for these location-based features.
442. [h3] 4.7.5. Performance and Scaling Observations
443. [h4] Current performance is adequate for the seeded dataset of 32 motorcycles and small user base, but several bottlenecks emerged during stress testing with larger datasets. Loading the entire motorcycle catalog for client-side filtering works smoothly with dozens of motorcycles but would cause noticeable delays with thousands. Migration to server-side pagination and filtering will be necessary before production launch.
444. [h4] Order queries currently fetch all related data (motorcycles, user information) without pagination, creating potential performance issues as order histories grow. Implementing cursor-based pagination for order histories will prevent memory issues and improve response times for users with extensive purchase histories.
445. [h4] Database query optimization through proper indexing on frequently searched fields (brand, category, price range) will become critical as the catalog grows. The current setup works adequately with 32 motorcycles but would require tuning for catalogs with thousands of listings.
446. [h3] 4.7.6. Security Considerations for Production
447. [h4] The current security implementation provides development-appropriate protection but requires significant hardening before production deployment. JWT signing keys currently reside in configuration files, creating exposure risk. Production requires externalized secret management through environment variables or dedicated secrets managers.
448. [h4] CORS configuration currently allows requests from any origin (*), necessary for development flexibility but unacceptable for production. Strict origin whitelisting limited to approved frontend domains must be implemented alongside other security headers (HSTS, CSP, X-Frame-Options).
449. [h4] Password hashing uses BCrypt with default cost factor, chosen without performance benchmarking under load. Production should tune cost factor balancing security strength against authentication latency requirements during peak traffic.
450. [h4] Input validation currently exists at the service layer but would benefit from additional request-level validation to provide defense-in-depth against malicious inputs. Rate limiting on authentication endpoints will prevent brute-force attacks on user accounts.
451. [h3] 4.7.7. Honest Assessment and Phase 3 Requirements
452. [h4] Phase 2 successfully validates that an integrated motorcycle platform can work—the technology choices prove sound, the architecture supports required workflows, and users can complete end-to-end journeys from discovery through purchase and service booking. This represents meaningful progress from Phase 1's theoretical designs to working software.
453. [h4] However, significant work remains before production readiness. The gap between "functional prototype" and "production system" encompasses not just missing features but fundamental operational maturity: deployment automation, monitoring and observability, incident response capabilities, backup and recovery procedures, and sustained operation under real load.
454. [h4] Phase 3 must prioritize features that directly impact user trust and platform viability: email verification establishes account legitimacy, payment gateway integration enables actual transactions, notification system improves engagement and transparency. The interactive map functionality with geospatial store location will complete the service booking experience. Beyond features, operational infrastructure—monitoring, logging, alerting, deployment pipelines—determines whether the system can operate reliably at scale.
455. [h4] The strategic deferrals in Phase 2 were appropriate decisions enabling focus on core validation. Now that the integrated approach proves viable, Phase 3 must address these gaps systematically while establishing operational practices supporting continuous improvement and reliable service.

459. [h4] Phase 2 successfully transformed MotoMarket from conceptual design into a functional full-stack platform that validates the integrated marketplace approach for the motorcycle industry. This represents substantial progress from Phase 1's wireframes and system diagrams to working software handling realistic workflows.

The three-tier architecture cleanly separates presentation, business logic, and data access layers, enabling independent evolution of each tier. Cloudinary's managed media service eliminated the need for custom image infrastructure while delivering professional-grade performance through CDN distribution. The order status state machine prevents business rule violations by enforcing valid status transitions, ensuring inventory updates only occur for properly processed orders.
462. [h4] The development process revealed important insights about transitioning from design to implementation. Phase 1's class diagrams accurately captured entity relationships but underestimated the complexity of cascade behaviors, transaction boundaries, and concurrent access patterns. API specifications provided essential contracts for frontend-backend integration but required refinement based on actual data flow requirements and pagination needs. The Figma prototypes successfully guided component development but didn't capture all necessary states—loading indicators, error messages, empty states—that emerged during implementation.

Email verification remains unimplemented, creating security vulnerabilities and account recovery challenges. Payment processing records transaction identifiers without actual gateway integration, deferring PCI compliance and financial reconciliation complexity. Interactive map functionality with geospatial store location will be implemented in Phase 3 to complete the service discovery experience.

The architecture accommodates planned features—event publication points exist for future notifications, transaction recording supports payment gateway integration, store data structures await location enhancement. Phase 2 establishes a maintainable foundation demonstrating technical viability while clearly identifying the path to production deployment.

The systematic progression from requirements analysis through architecture design to working implementation reflects professional software engineering practices.

By unifying discovery, commerce, service coordination, and community features within a single system, MotoMarket reduces transaction costs, improves information flow, and creates new value propositions for multiple stakeholder groups. The implementation offers adaptable patterns and architectural blueprints for other domains facing similar fragmentation challenges—automotive services, real estate, healthcare—where coordinated workflows across traditionally separate functions could deliver substantial user value.

470. [h3] 5.2.1. Critical User-Facing Features
471. [n] Email Verification and Account Security
472. [h4] Implementing email verification requires SMTP server configuration (SendGrid or AWS SES), verification token generation with time-based expiration, HTML email templates with branding and clear calls-to-action, activation workflow endpoints handling token validation and account activation, and UI flows for resending verification emails and handling expired tokens. Additionally, password recovery workflows require similar infrastructure with secure reset tokens, validation of reset requests, and password change forms with strength requirements. These features gate production launch as they establish account legitimacy and enable self-service recovery.
473. [n] Payment Gateway Integration
474. [h4] Integrating Stripe for production payment processing requires: Stripe SDK configuration with API keys and webhook secrets, checkout session creation endpoints generating payment intents, webhook endpoints receiving asynchronous payment confirmations with idempotent processing to handle duplicate deliveries, refund API integration for order cancellations and returns, payment reconciliation dashboards for administrative oversight, and comprehensive error handling for payment failures with clear user feedback. Testing requires extensive work with Stripe's test mode, simulating various payment scenarios (successful payments, declined cards, network failures, webhook delays), and ensuring proper order state transitions based on payment results.
475. [n] Real-Time Notification System
476. [h4] Building notification infrastructure requires: message queue implementation (RabbitMQ or AWS SQS) for reliable delivery, notification preference management allowing users to control email/SMS/push settings per event type, message template system with variable substitution and multi-language support, delivery mechanism integrations (email via SendGrid, SMS via Twilio, push notifications via Firebase), delivery tracking recording sent notifications with status updates, retry logic with exponential backoff for failed deliveries, and unsubscribe mechanisms respecting user preferences. Priority events include order status changes, booking confirmations, forum replies, and promotional announcements.
477. [n] Interactive Map and Location Features
478. [h4] Completing location-based services requires: Google Maps or Mapbox integration for interactive map displays, geolocation API usage requesting user position with appropriate permission handling, store location visualization with custom markers and information windows, nearest-store calculation using geospatial distance algorithms (Haversine formula), route planning integration showing directions from user location to selected stores, and responsive map components working across desktop and mobile devices. The backend store data structure already supports latitude/longitude storage, establishing the foundation for these features.
479. [h3] 5.2.2. Operational Infrastructure
480. [n] Deployment Automation and Environment Management
481. [h4] Production deployment requires: containerization with Docker creating consistent runtime environments, Docker Compose configurations for local development matching production architecture, container orchestration with Kubernetes enabling scaling and self-healing, CI/CD pipelines using GitHub Actions or GitLab CI automating build/test/deploy workflows, environment-based configuration management separating development/staging/production settings, secrets management with AWS Secrets Manager or HashiCorp Vault protecting sensitive credentials, and blue-green deployment strategies enabling zero-downtime updates.
482. [n] Monitoring and Observability
483. [h4] Reliable operation demands: centralized logging with ELK stack (Elasticsearch, Logstash, Kibana) or CloudWatch aggregating logs from all services, structured logging with consistent formats enabling effective querying, metrics collection via Micrometer and Prometheus tracking request rates/response times/error rates, distributed tracing with Jaeger or AWS X-Ray following requests across service boundaries, dashboards with Grafana visualizing system health and performance trends, alerting rules triggering notifications for critical conditions (high error rates, database connection failures, elevated latency), and on-call rotation procedures for incident response.
484. [n] Security Hardening
485. [h4] Production security requires: JWT refresh token implementation with longer-lived refresh tokens and shorter access tokens, rate limiting on authentication endpoints preventing brute-force attacks, comprehensive input validation using Spring's @Valid annotations, HTTPS enforcement redirecting all HTTP traffic, security headers configuration (HSTS, CSP, X-Frame-Options, X-Content-Type-Options), CORS tightening restricting origins to approved frontend domains, dependency vulnerability scanning with tools like Snyk or Dependabot, regular penetration testing identifying security weaknesses, and security audit logging tracking sensitive operations.
486. [h3] 5.2.3. Performance Optimization and Scalability
487. [n] Database Optimization
488. [h4] Scaling database performance requires: indexing strategy implementation on frequently queried fields (motorcycle brand/category/price, order status/creation date, user email/username), query optimization through JPA fetch strategies (eager vs lazy loading) and query plan analysis, connection pooling tuning for optimal throughput under load, read replica configuration for separating read-heavy operations from writes, and database migration to managed services (AWS RDS) with automatic backups and high availability.
489. [n] Caching Strategy
490. [h4] Implementing caching requires: Redis deployment for distributed caching across application instances, catalog caching storing motorcycle listings with time-based expiration, user session caching reducing database queries for authenticated requests, cache invalidation strategies ensuring consistency when data changes, and cache warming for frequently accessed data during deployment.
491. [n] Frontend Performance
492. [h4] Optimizing frontend performance requires: server-side pagination replacing client-side filtering for catalog listings, lazy loading images deferring off-screen content, code splitting creating smaller initial bundles, Progressive Web App (PWA) implementation enabling offline functionality and app-like experience, and CDN deployment for static assets reducing latency globally.
493. [h3] 5.2.4. Enhanced Features and Business Logic
494. [n] Inventory Management
495. [h4] Advanced inventory features include: low stock alerts notifying administrators when quantities fall below thresholds, stock reservation during checkout holding inventory temporarily for active purchases, inventory forecasting predicting reorder needs based on sales patterns, and supplier integration automating purchase orders when stock depletes.
496. [n] Analytics and Reporting
497. [h4] Business intelligence features include: sales analytics tracking revenue by brand/category/time period, user behavior analysis identifying popular motorcycles and conversion funnels, service booking trends informing capacity planning, forum engagement metrics measuring community health, and administrative dashboards visualizing key performance indicators.
498. [n] Search Enhancement
499. [h4] Improving search capabilities requires: Elasticsearch integration enabling full-text search across motorcycle descriptions and features, autocomplete suggestions appearing as users type, search result ranking using relevance scoring and user behavior signals, and faceted search providing count previews for each filter option.
